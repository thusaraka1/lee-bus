# Goal Description

The objective is to design, develop, and integrate a comprehensive **Smart Passenger Reporting and Bus Monitoring System**. This system will combine real-time IoT hardware (already partially developed for ESP32) with a cloud backend, an intelligent analytics engine, an admin dashboard, and a friction-free QR-based passenger reporting web app.

This implementation plan outlines the architecture, technology stack, data flow, and a phased execution strategy to seamlessly integrate all requested features into a robust, real-time hybrid system.

---

## System Architecture & Technology Stack

To achieve the combination of real-time monitoring, intelligent processing, and scalable user interfaces, the following architecture is proposed:

### 1. IoT Hardware (Edge Device - ESP32)
*   **Microcontroller:** ESP32 (Existing code uses MPU6050, NEO-6M GPS, HW-484 Mic, SIM7670C LTE).
*   **Connectivity:** MQTT over LTE (HiveMQ Cloud is currently used). 
*   **Offline Support:** Add an SD Card module or use ESP32 internal SPIFFS/LittleFS to store JSON payloads locally when the MQTT connection fails, syncing them when connectivity is restored.

### 2. Cloud Infrastructure & Backend System
*   **Message Broker:** HiveMQ Cloud (Currently implemented in firmware).
*   **Backend Server:** Node.js with Express (or Python FastAPI for better data science/ML integration). This will consume MQTT messages, process the data, and expose REST APIs.
*   **Database:** 
    *   **Firebase Realtime Database / Firestore:** For live bus tracking, real-time alerts, and fast syncing to the Admin Dashboard.
    *   *(Alternative)* **PostgreSQL + TimescaleDB:** If long-term analytics and heavy time-series data storage are required.
*   **Intelligent Detection Engine:** A background worker process (Node.js/Python) that evaluates streaming data against thresholds (e.g., sudden G-force spikes from MPU6050 = hard braking/accident; Speed > limit = Overspeeding).

### 3. Passenger QR Web App
*   **Frontend:** Next.js (React) or lightweight Vanilla HTML/JS with Tailwind CSS. Mobile-first design.
*   **QR Strategy:** Dynamic QR codes generated by the Admin system upon bus registration. The QR encodes a URL (e.g., `https://app.com/report?busId=B123`) that automatically opens the web form with pre-filled context.

### 4. Admin Dashboard
*   **Frontend:** Next.js or React (Vite).
*   **Mapping:** Leaflet.js or Google Maps API for live fleet tracking.
*   **Bus Management & QR Generation:** Dedicated section to register new buses (auto-generating a unique QR code) and a repository to view/download QR codes for all existing registered buses.
*   **Analytics:** Recharts or Chart.js for data visualization (complaints, driver scores, risk prediction).

## Phased Implementation Plan

We will build the system in a logical, step-by-step manner to ensure each layer is stable before adding the next.

### Phase 1: Hardware Hardening & Cloud Ingestion
*   **Hardware Offline Storage:** Update the ESP32 firmware to write data to local storage (SD/SPIFFS) if the SIM7670C module loses connection. Implement a routine to publish stored data once reconnected.
*   **Cloud Ingestion Setup:** Create a Node.js/Python background service that subscribes to the HiveMQ `lee_bus/sensors` topic.
*   **Database Integration:** Configure the ingestion service to parse incoming JSON and save it to Firebase Firestore/Realtime Database.

### Phase 2: Intelligent Backend & Automated Alerts
*   **Unsafe Behavior Algorithms:** Implement logic in the backend to detect:
    *   *Overspeeding:* Current GPS speed > Route limit.
    *   *Sudden Braking/Accident:* Rapid changes in Accelerometer/Gyro (X, Y, Z axes).
    *   *Environmental:* Mic dB > Threshold or Temp anomalies.
*   **Automated Alerts:** Integrate an Email (SendGrid) or SMS (Twilio) service to instantly notify authorities of critical detections.

### Phase 3: Admin Dashboard (Real-Time UI & Bus Management)
*   **Live Tracking:** Build the map interface subscribing to Firebase for real-time location updates.
*   **Bus Registration & QR System:** Create the interface to add buses. Upon creation, automatically generate a unique QR code image (using a library like `qrcode.react`). Provide a view/download gallery for existing buses.
*   **Fleet & Driver Overview:** Create views linking specific buses to drivers. Calculate a "Driver Safety Score" dynamically based on IoT anomaly counts (speeding events) and passenger complaints.
*   **Risk Prediction Analytics:** Build charts showing historical data to flag high-risk drivers and problematic routes.

### Phase 4: Passenger QR Reporting System
*   **QR Web App:** Develop the mobile web app for passengers. No login required.
*   **Complaint Ingestion:** When a passenger submits a report, the backend logs it, associates it with the active driver of that bus, and pushes an immediate notification to the Admin Dashboard.

### Phase 5: System Testing & Evaluation
*   **Hybrid Testing:** Verify that physical hardware actions (tilting the MPU6050, moving the GPS) reflect accurately on the Admin map and trigger alerts if thresholds are exceeded.
*   **Evaluation:** Assess response times from the point of data capture to dashboard display.
